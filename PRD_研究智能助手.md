# PRD：多模型协同研究智能助手

## 1. 产品概述

### 1.1 产品名称
研究智能助手（暂定名）

### 1.2 产品定位
面向研究者的多模型协同研究工作台。系统围绕“选题跟踪、论文获取、文献理解、研究思路比较、实验执行与加速”构建完整闭环，让研究者把精力集中在研究 taste、问题判断和关键洞察上，把高频、重复、可并行的工作交给多个大模型与自动化流程协同完成。

### 1.3 一句话定义
这是一个以研究者为核心、以多模型分工为方法、以研究深度和迭代速度同时提升为目标的 AI 研究伙伴。

## 2. 背景与机会

当前研究工作存在以下痛点：

1. 新论文更新过快，研究者很难持续、高质量地跟踪 arXiv 与顶会论文。
2. 论文获取、归档、摘要整理、方法对比等环节耗时高且重复性强。
3. 单一大模型在长链条研究任务中往往存在能力边界，难以兼顾总结质量、推理深度、代码实验与成本控制。
4. 研究者真正稀缺的不是“读不到论文”，而是无法把有限时间优先投入到高价值判断：哪些方向值得追，哪些方法只是表面创新，哪些实验能快速验证想法。
5. 研究节奏常常被低价值机械劳动拖慢，导致“广度”和“深度”难以兼得。

随着不同模型在不同能力维度上逐渐分化，构建“模型组合 + 自动化工作流 + 人的判断闭环”的个人研究系统，具备较强现实价值。

## 3. 产品愿景

打造一个真正服务于个人研究能力增长的智能助手，不替代研究者做最终判断，而是成为研究者的“研究操作系统”：

1. 帮研究者更快发现值得关注的问题与论文。
2. 帮研究者更系统地理解不同论文之间的差异、路线和空白点。
3. 帮研究者更低成本地组织实验、调用模型、并行推进。
4. 让研究者把主要认知资源投入到 taste、洞察、问题定义和关键决策上。

## 4. 产品目标

### 4.1 核心目标

1. 建立多选题、多来源的论文跟踪能力。
2. 自动完成论文下载、清洗、结构化整理与知识归档。
3. 自动生成高质量文献总结、方法对比、研究脉络图谱。
4. 基于多个模型的能力分工，提升复杂任务完成质量与效率。
5. 支持研究者发起实验任务、并行执行、统一回收结果。
6. 形成“跟踪 -> 理解 -> 比较 -> 假设 -> 实验 -> 复盘”的研究闭环。

### 4.2 业务目标

1. 显著降低研究信息整理时间。
2. 缩短从“发现论文”到“形成可行动研究思路”的周期。
3. 提高研究者在多个选题上并行推进的能力。
4. 构建个人长期可复用的研究知识资产。

## 5. 非目标

以下内容不作为第一阶段重点：

1. 面向大众用户的开放平台。
2. 完整替代研究者做选题判断或学术判断。
3. 端到端全自动发表论文系统。
4. 大规模社交协作社区能力。
5. 与所有论文网站和实验平台的深度一次性打通。

## 6. 目标用户

### 6.1 核心用户

1. 有持续研究任务的个人研究者。
2. 希望建立个人研究工作流的硕博学生。
3. 同时跟踪多个方向、需要高频阅读和实验的算法工程师或独立研究者。

### 6.2 用户特征

1. 有明确研究目标，但时间和精力有限。
2. 愿意使用 AI 提升效率，但不希望被“自动生成内容”淹没。
3. 更关注高质量思考，而不是表面上的自动化炫技。
4. 对输出的可追溯性、可信度和可复盘性要求较高。

## 7. 核心用户价值

系统为用户提供三类核心价值：

1. 信息处理提速：自动跟踪、筛选、下载、解析、归档论文。
2. 认知组织增深：自动总结、对比、归纳研究路线，帮助形成判断。
3. 实验执行加速：将代码、实验、验证任务交给不同模型和执行器并行完成。

最终价值不是“替用户思考”，而是“把用户的思考放在最值钱的位置上”。

## 8. 核心理念

### 8.1 人机分工原则

1. 人负责：研究 taste、方向判断、问题定义、关键假设、最终结论。
2. 模型负责：信息收集、内容整理、模式比较、实验实现、结果汇总。

### 8.2 多模型协同原则

不同模型承担不同难度与风格的任务：

1. 轻量模型：摘要提取、格式整理、标签分类、初步去重。
2. 文本组织型模型：总结、改写、结构化表达、汇报输出。
3. 强推理模型：研究问题拆解、方法差异分析、潜在创新点挖掘。
4. 强代码模型：实验代码编写、调试、脚本生成、批量执行。
5. 审核模型：交叉检查事实一致性、输出质量与实验结论。

### 8.3 系统目标不是“全自动”

系统应优先支持“可控自动化”而非“黑盒自动化”，让研究者始终可以：

1. 查看来源与证据。
2. 干预模型分工与工作流。
3. 修改结论、补充偏好、设定研究 taste。
4. 决定是否进入实验与投入更多算力。

## 9. 典型使用场景

### 9.1 场景一：跟踪多个研究选题
用户预设多个主题，例如多模态推理、Agent、检索增强、科学发现等。系统每天或每周自动扫描 arXiv、会议官网、OpenReview 等来源，收集相关新论文并完成初筛。

### 9.2 场景二：快速形成文献图谱
用户打开某个方向页面，可以直接看到：

1. 最近新增论文列表。
2. 每篇论文的核心贡献、方法摘要与实验亮点。
3. 与已有代表性论文的差异点。
4. 该方向的主要技术路线与空白点。

### 9.3 场景三：围绕一个具体 idea 做研究比较
用户输入一个研究想法或问题，系统自动检索相关论文，比较已有路线，并产出：

1. 已有人做过什么。
2. 哪些问题尚未被充分解决。
3. 可能的切入点有哪些。
4. 哪些实验最适合作为低成本验证。

### 9.4 场景四：把实验工作并行交给模型
用户确认方向后，系统把不同子任务分发给不同模型或执行器，例如：

1. 一个模型负责复现实验框架。
2. 一个模型负责写数据处理脚本。
3. 一个模型负责生成 ablation 方案。
4. 一个模型负责汇总日志和结果。

## 10. 用户流程

### 10.1 主流程

1. 用户配置研究主题、关键词、会议源和个人偏好。
2. 系统定期抓取新论文并建立候选池。
3. 系统下载 PDF、抽取元数据、解析正文并结构化存储。
4. 多模型协同生成摘要、方法拆解、亮点与局限分析。
5. 系统对论文进行聚类、去重、路线归类和关键论文关联。
6. 用户查看方向看板，筛选重点论文与重点问题。
7. 用户发起某个专题分析或研究假设分析。
8. 系统生成研究对比报告、创新空白提示和实验建议。
9. 用户选择一部分任务进入实验阶段。
10. 系统调度不同模型与实验执行环境并行运行。
11. 系统汇总结果，形成实验日报、失败原因和下一步建议。
12. 用户基于结果做判断，沉淀新的研究笔记与偏好。

## 11. 产品功能需求

### 11.1 研究主题管理

功能说明：

1. 支持创建多个研究主题。
2. 每个主题支持配置关键词、排除词、优先作者、目标会议、时间范围。
3. 支持记录用户对该主题的研究目标、判断标准和个人 taste。

需求要点：

1. 支持主题分组与优先级排序。
2. 支持主题级别通知策略。
3. 支持保存用户定义的“值得关注信号”。

### 11.2 论文源跟踪与抓取

功能说明：

1. 跟踪 arXiv 分类、关键词检索结果、会议官网、OpenReview 等来源。
2. 定时拉取新增论文。
3. 自动识别重复论文和版本更新。

需求要点：

1. 支持按来源、时间、主题、热度过滤。
2. 支持记录抓取时间、来源链接、版本号。
3. 支持失败重试与抓取日志。

### 11.3 论文下载与解析

功能说明：

1. 自动下载 PDF 与补充材料。
2. 抽取标题、作者、摘要、发布日期、代码链接、机构等元数据。
3. 解析正文结构，如引言、方法、实验、结论。

需求要点：

1. 支持 PDF 解析失败回退策略。
2. 支持图表、公式、参考文献的基础识别。
3. 支持本地知识库归档和全文检索。

### 11.4 文献总结与结构化理解

功能说明：

1. 自动输出不同长度的论文摘要。
2. 提取研究问题、核心方法、关键创新、实验结论、局限性。
3. 生成统一结构的论文卡片。

需求要点：

1. 支持摘要风格模板，如速读版、研究版、汇报版。
2. 支持中英文双语输出。
3. 支持附带证据引用，降低幻觉风险。

### 11.5 研究思路比较与路线分析

功能说明：

1. 自动比较多篇论文的研究问题、技术路线、实验设计和适用场景。
2. 识别路线差异、共性假设、评价盲区与潜在空白。
3. 形成专题分析报告。

需求要点：

1. 支持“给定论文集合”的深度比较。
2. 支持“围绕一个问题”的跨论文比较。
3. 支持输出研究脉络图、方法树或时间线。

### 11.6 多模型任务编排

功能说明：

1. 根据任务类型自动选择适合的模型。
2. 支持用户自定义模型角色分工。
3. 支持串行、并行、评审式和辩论式工作流。

示例分工：

1. 千问负责文字组织、总结与中文表达优化。
2. Codex / Claude 负责复杂分析、代码实现、实验拆解。
3. 低成本模型负责批量初筛、标签分类、候选摘要。
4. 高质量模型负责最终报告审校与关键结论把关。

需求要点：

1. 支持模型能力画像与成本记录。
2. 支持失败重试、模型替补、结果投票。
3. 支持用户查看每个结论由哪个模型产出。

### 11.7 实验任务生成与执行

功能说明：

1. 用户可从论文、问题或研究假设直接发起实验任务。
2. 系统自动拆解为数据准备、环境准备、脚本编写、运行、评估等步骤。
3. 支持并行调度多个实验任务。

需求要点：

1. 支持实验模板与复用。
2. 支持日志采集、报错归因、结果总结。
3. 支持对实验结果自动生成复盘报告。

### 11.8 知识沉淀与个人研究记忆

功能说明：

1. 把论文总结、比较结论、实验结果、用户批注统一沉淀为个人知识库。
2. 记录用户长期偏好、常见判断框架和研究兴趣变化。
3. 为后续分析提供个性化上下文。

需求要点：

1. 支持笔记挂载到论文、主题、实验与结论。
2. 支持反向查询，例如“过去两个月我在哪些方向最活跃”。
3. 支持将用户显式反馈纳入后续推荐与排序。

## 12. 关键页面与信息架构

### 12.1 首页仪表盘

展示内容：

1. 今日/本周新增论文数量。
2. 各研究主题最新动态。
3. 待处理高优先级论文。
4. 运行中的实验任务。
5. 最近形成的研究洞察与结论。

### 12.2 研究主题页

展示内容：

1. 主题描述与研究目标。
2. 论文流与重点论文卡片。
3. 技术路线图。
4. 主题级研究笔记。
5. 相关实验与阶段结论。

### 12.3 论文详情页

展示内容：

1. 基础元数据与原始链接。
2. 自动摘要与结构化卡片。
3. 与相似论文的比较。
4. 用户批注与引用证据。
5. 可发起的研究分析或实验操作。

### 12.4 专题比较页

展示内容：

1. 多篇论文对比矩阵。
2. 方法差异和共同点。
3. 潜在研究机会。
4. 推荐实验方案。

### 12.5 实验中心

展示内容：

1. 实验队列。
2. 任务状态。
3. 运行日志。
4. 指标结果。
5. 失败分析与下一步建议。

## 13. 模型协同策略

### 13.1 角色分层

1. Scout 模型：负责低成本批量检索、筛选、分类。
2. Writer 模型：负责摘要、整理、汇报、文档产出。
3. Analyst 模型：负责比较、推理、创新点分析。
4. Builder 模型：负责代码、实验、脚本与执行。
5. Judge 模型：负责评审、纠错、交叉验证。

### 13.2 调度原则

1. 简单任务优先使用低成本模型。
2. 高价值、低容错任务优先使用高质量模型。
3. 复杂任务先拆解再分发，避免单模型长链条失败。
4. 关键结论应至少经过一次复核或多模型交叉校验。

### 13.3 用户可配置项

1. 每类任务默认模型。
2. 每日/每周 token 或费用预算。
3. 是否启用交叉评审。
4. 对速度、质量、成本的偏好权重。

## 14. 数据与知识结构

建议围绕以下对象建模：

1. Topic：研究主题。
2. Paper：论文实体。
3. PaperVersion：论文版本。
4. Summary：摘要与结构化理解结果。
5. Comparison：论文比较结果。
6. Insight：研究洞察。
7. Experiment：实验任务。
8. Artifact：代码、日志、图表、结果文件。
9. UserPreference：用户偏好与 taste。
10. WorkflowRun：一次模型协同运行记录。

## 16. 产品指标

### 16.1 效率指标

1. 从论文发布到进入用户待处理列表的延迟。
2. 单篇论文完成下载、解析与摘要的耗时。
3. 从用户提出问题到生成专题分析报告的耗时。

### 16.2 质量指标

1. 摘要与原文一致性评分。
2. 论文分类与主题归属准确率。
3. 研究比较报告的用户满意度。
4. 实验任务成功执行率。

### 16.3 用户价值指标

1. 每周被用户真正深入阅读的论文数。
2. 每月形成的有效研究洞察数。
3. 从论文跟踪到实验验证的转化率。
4. 用户在系统内积累的个人研究资产规模。

## 17. 风险与挑战

### 17.1 模型幻觉与错误总结

风险：
模型可能错误归纳论文贡献、夸大实验效果或编造细节。

应对：

1. 保留原文引用与证据定位。
2. 对关键结论启用复核模型。
3. 对高风险任务要求用户确认。

### 17.2 PDF 解析质量不稳定

风险：
不同论文排版复杂，可能影响段落、公式与图表解析。

应对：

1. 设计多解析器回退机制。
2. 对关键段落支持人工校对。
3. 优先保证结构化摘要而非完美全文还原。

### 17.3 多模型协同成本过高

风险：
多个模型并行运行可能带来较高调用成本。

应对：

1. 引入任务分级与预算控制。
2. 先用低成本模型过滤，再调用高质量模型。
3. 对重复任务做缓存与复用。

### 17.4 实验自动化难度高

风险：
真实研究实验常涉及复杂依赖、数据、环境和调参。

应对：

1. MVP 阶段先支持轻量实验和脚本辅助。
2. 将实验拆成更细颗粒度任务。
3. 强化日志、失败诊断和人工接管能力。

## 18. 产品原则

1. 优先服务研究判断，而不是堆叠功能。
2. 优先支持可追溯与可控，而不是黑盒自动化。
3. 优先把高频低价值工作自动化，把高价值认知工作留给用户。
4. 优先支持个人长期积累，而不是一次性输出。
5. 优先用多模型分工解决真实问题，而不是为了“多模型”而多模型。

## 19. 版本规划建议

### V0.1：研究信息流基础版

目标：
先跑通“跟踪 -> 下载 -> 摘要 -> 归档”。

重点功能：

1. 研究主题管理。
2. arXiv 抓取。
3. PDF 下载与元数据解析。
4. 论文摘要卡片。
5. 本地知识库存档。

### V0.2：研究比较版

目标：
从“读论文”升级到“做判断”。

重点功能：

1. 多篇论文对比。
2. 研究路线分析。
3. 用户批注与 insight 沉淀。
4. 多模型角色编排。

### V0.3：实验加速版

目标：
从“理解论文”升级到“推进研究”。

重点功能：

1. 实验任务拆解。
2. 脚本生成与运行。
3. 并行实验调度。
4. 实验结果复盘与建议。

## 20. 结语

这个产品的核心，不是造一个“会读论文的机器人”，而是构建一套能持续增强研究者能力的个人研究系统。它应该帮助用户把更多时间投入到真正决定研究上限的部分：问题选择、判断标准、创新 taste 和深度思考；同时把论文跟踪、文献整理、实验执行这些可自动化部分最大程度提速。

如果这个方向做对了，用户获得的不是一次性的效率工具，而是一套随着时间积累越来越强的研究操作系统。
