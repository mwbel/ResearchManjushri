## 2. 项目目标拆解

围绕产品目标，先拆成 4 个一级系统：

1. 论文跟踪系统：负责发现、抓取、下载、去重、更新。
2. 文献理解系统：负责解析、总结、对比、知识沉淀。
3. 多模型编排系统：负责模型路由、任务拆解、并行执行、结果复核。
4. 实验执行系统：负责把研究想法转为实验任务，并记录过程与结果。

对应第一阶段的核心闭环是：

1. 配置研究主题。
2. 自动跟踪 arXiv / 顶会来源。
3. 下载论文并生成结构化卡片。
4. 对多篇论文做比较和专题分析。
5. 将结果沉淀到个人知识库。

## 3. 技术架构总览

建议采用“单人可控、逐步演进”的分层架构，而不是一开始就做复杂微服务。

### 3.1 架构分层

1. 前端层：研究工作台 UI，负责主题管理、论文浏览、比较报告、实验中心。
2. 应用层：业务 API，负责主题管理、工作流触发、结果查询、用户配置。
3. 工作流层：任务编排引擎，负责抓取、解析、总结、比较、实验等异步流程。
4. 模型层：统一模型网关，负责路由不同大模型与提示词模板。
5. 数据层：关系数据库 + 向量索引 + 文件存储。
6. 集成层：arXiv、OpenReview、会议官网、PDF 解析器、代码仓库、实验环境。

### 3.2 推荐技术选型

#### 前端

1. `Next.js`
2. `TypeScript`
3. `Tailwind CSS`
4. `shadcn/ui` 或简洁组件库

原因：

1. 适合快速搭建研究工作台。
2. 服务端渲染与 API 路由灵活。
3. 后续做搜索、详情页和仪表盘都比较顺手。

#### 后端 / API

1. `FastAPI` 或 `Next.js API + Python worker`
2. 推荐主业务 API 用 `FastAPI`
3. 推荐模型、论文处理、实验相关逻辑尽量放 Python 生态

原因：

1. 论文处理、数据清洗、LLM 编排、实验执行通常更适合 Python。
2. 后续接 LangGraph、抓取器、PDF 解析、科研脚本更自然。

#### 任务编排

1. 早期：`Celery` / `RQ` / 轻量队列
2. 后续：`LangGraph` 或自定义工作流 DAG

原因：

1. 一开始不要过度依赖复杂 Agent 框架。
2. 先把任务拆清楚，再决定是否引入更复杂的有状态工作流。

#### 数据存储

1. `PostgreSQL`：存元数据、主题、任务、实验、模型调用记录。
2. `pgvector` 或独立向量库：存论文 embedding 与语义检索索引。
3. 本地文件系统 / 对象存储：存 PDF、解析结果、实验产物。

#### 模型接入

1. 自建统一 `LLM Gateway`
2. 支持 OpenAI / Claude / Qwen / 其他模型供应商
3. 按任务类型做路由而不是写死在业务代码里

#### 论文解析

1. `GROBID`、`PyMuPDF`、`pdfplumber`、OCR 回退
2. 先保证“可用摘要结构化”，再追求“完美论文还原”

#### 实验执行

1. 本地执行器作为第一阶段起点
2. 后续扩展到远程机器 / GPU 节点
3. 每个实验任务有独立运行目录和日志

2. arXiv 抓取器。

7. 多模型路由与结果复核。
8. 知识库检索。

3. 实验模块价值很高，但应建立在前面结构已经稳定的基础上。

## 10. 风险控制清单

### 技术风险

1. PDF 解析质量不稳定。
2. 不同模型输出风格差异大。
3. 多任务编排容易失控，调试难度高。
4. 实验执行依赖环境复杂。

应对：

1. 优先把流程切小。
2. 每个模块先做最小可用版本。
3. 关键步骤都保留中间产物。
4. 关键结论增加复核机制。

### 项目风险

1. 一开始目标太大，容易长时间没有可见成果。
2. 同时做 UI、抓取、模型编排、实验系统，精力容易分散。

应对：

1. 先做“论文流 + 卡片 + 比较”这个最核心闭环。
2. 不要在第一阶段过早追求完美 agent 化。
3. 每两周必须产出一个可演示版本。
